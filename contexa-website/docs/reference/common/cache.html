Cache Service

Multi-tier caching infrastructure supporting LOCAL (Caffeine), REDIS, and HYBRID (L1 Caffeine + L2 Redis) modes with domain-specific TTL configuration and distributed cache invalidation via Redis Pub/Sub.

ContexaCacheService Interface

Java 
public interface ContexaCacheService {

    // Get with cache-aside pattern: check cache, fallback to loader
    <T> T get(String key, Supplier<T> loader,
              TypeReference<T> typeRef, String domain);

    // Put a value into cache for the given domain
    <T> void put(String key, T value, String domain);

    // Invalidate by key (supports wildcard prefix with *)
    void invalidate(String key);

    // Invalidate all cached entries
    void invalidateAll();

    // Invalidate local cache only (for distributed invalidation events)
    void invalidateLocalOnly(String key);

    // Get the configured cache type
    ContexaCacheProperties.CacheType getCacheType();
}

All values are serialized to JSON via Jackson ObjectMapper. The TypeReference parameter ensures correct deserialization of generic types.

Cache Architecture

Text 
                    ContexaCacheService
                           |
              +------------+------------+
              |                         |
   LocalContexaCacheService    RedisContexaCacheService
              |                         |
   [Caffeine L1 only]         [Supports all 3 modes]
                                        |
                          +-------------+-------------+
                          |             |             |
                       LOCAL         REDIS         HYBRID
                    (Caffeine)    (Redis only)   (L1 + L2)
                                                     |
                                               Pub/Sub invalidation

Cache Modes

ModeL1 (Caffeine)L2 (Redis)Pub/SubUse Case
LOCAL Yes No No Single-instance deployments, development
REDIS No Yes No Multi-instance with shared cache, no local cache needed
HYBRID Yes Yes Yes Multi-instance with local cache for latency + Redis for consistency

AbstractContexaCacheService

Base class for both implementations. Manages domain-specific Caffeine caches using a ConcurrentHashMap<String, Cache<String, String>> and provides TTL resolution by domain.

Java 
public abstract class AbstractContexaCacheService
        implements ContexaCacheService {

    protected final ContexaCacheProperties properties;
    protected final ObjectMapper objectMapper;
    protected final ConcurrentHashMap<String, Cache<String, String>>
            domainCaches = new ConcurrentHashMap<>();
    protected Cache<String, String> defaultLocalCache;

    // Creates or retrieves a domain-specific Caffeine cache
    protected Cache<String, String> getOrCreateDomainCache(String domain);

    // Builds Caffeine cache with configured maxSize, TTL, and stats recording
    protected Cache<String, String> buildLocalCache(int ttlSeconds);

    // Resolves local TTL for a domain from properties
    protected int getLocalTtl(String domain);
}

LocalContexaCacheService

Pure in-memory caching using Caffeine. Suitable for standalone deployments.

  • get() — Checks domain-specific Caffeine cache, deserializes from JSON, loads from supplier on miss, caches the result
  • put() — Serializes value to JSON and stores in domain-specific Caffeine cache
  • invalidate() — Supports wildcard prefix invalidation (e.g., users:*)
  • invalidateAll() — Clears all domain caches and the default cache
  • invalidateLocalOnly() — Identical to invalidate() since there is no remote cache

RedisContexaCacheService

Supports all three cache modes. In HYBRID mode, it implements a two-tier lookup with L1 backfill and Pub/Sub-based distributed invalidation.

Read Path (HYBRID mode)

Text 
get(key, loader, typeRef, domain)
    |
    v
[1] Check L1 (Caffeine) --> HIT? return deserialized value
    |
    v
[2] Check L2 (Redis) --> HIT? backfill to L1, return value
    |
    v
[3] Call loader.get() --> cache in L1 + L2, return value

Write Path (HYBRID mode)

Text 
put(key, value, domain)
    |
    +---> Serialize to JSON
    +---> Store in L1 (Caffeine) with domain-specific local TTL
    +---> Store in L2 (Redis) with domain-specific Redis TTL
          Key prefix: contexa:cache:{key}

Invalidation Path (HYBRID mode)

Text 
invalidate(key)
    |
    +---> Invalidate L1 (local Caffeine, supports wildcard)
    +---> Invalidate L2 (Redis DELETE or SCAN+DELETE for wildcard)
    +---> Publish invalidation event via Redis Pub/Sub
          Channel: contexa:cache:invalidation

Redis key scanning uses SCAN with a count hint of 100 to avoid blocking the Redis server during wildcard invalidation.

ContexaCacheInvalidationListener

Listens for distributed cache invalidation events on the configured Redis Pub/Sub channel and invalidates the local (L1) cache only.

Java 
public class ContexaCacheInvalidationListener {

    private final ContexaCacheService cacheService;
    private final ContexaCacheProperties properties;

    public void handleMessage(String message, String channel) {
        // Validates channel matches configured Pub/Sub channel
        // "INVALIDATE_ALL" -> invalidateLocalOnly("*")
        // Other messages -> invalidateLocalOnly(key)
    }
}

Configuration Properties

YAML 
contexa:
  cache:
    type: REDIS             # LOCAL | REDIS | HYBRID
    local:
      max-size: 1000        # Maximum entries per domain cache
      default-ttl-seconds: 60
    redis:
      default-ttl-seconds: 300
      key-prefix: "contexa:cache:"
    pubsub:
      enabled: true
      channel: "contexa:cache:invalidation"
    domains:
      users:
        local-ttl-seconds: 3600
        redis-ttl-seconds: 3600
      roles:
        local-ttl-seconds: 14400
        redis-ttl-seconds: 14400
      permissions:
        local-ttl-seconds: 28800
        redis-ttl-seconds: 28800
      groups:
        local-ttl-seconds: 14400
        redis-ttl-seconds: 14400
      policies:
        local-ttl-seconds: 30
        redis-ttl-seconds: 300
      hcad:
        local-ttl-seconds: 86400
        redis-ttl-seconds: 86400

Domain TTL Reference

DomainLocal TTLRedis TTLRationale
users1 hour1 hourUser data changes moderately often
roles4 hours4 hoursRole definitions change infrequently
permissions8 hours8 hoursPermission definitions are relatively stable
groups4 hours4 hoursGroup membership changes at moderate frequency
policies30 seconds5 minutesPolicies need short local TTL for quick updates; Redis TTL is longer since Pub/Sub handles invalidation
hcad24 hours24 hoursHCAD model data is recomputed on a daily cycle

Mode Selection

The cache mode is determined by the contexa.cache.type property and the availability of Redis on the classpath:

  • LOCAL — LocalContexaCacheService is auto-configured. No Redis dependency required.
  • REDIS — RedisContexaCacheService is auto-configured with Redis-only operations. Requires spring-boot-starter-data-redis.
  • HYBRID — RedisContexaCacheService is auto-configured with both L1 (Caffeine) and L2 (Redis) tiers. The ContexaCacheInvalidationListener is also registered for Pub/Sub invalidation.
Configuration Reference
For cache application.yml properties (mode, TTL, max sizes, L1/L2 tiers), see Infrastructure Configuration — covers ContexaCacheProperties and SecurityRedisProperties.
